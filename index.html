Let's talk structure

Reading and tab order are important

Designers and product managers often talk about how a user's eye will flow through the site, but when it comes to development this conversation is often left out. Your structure should follow the site's reading order.

  • If your structure flows correctly the user should easily be able to use the tab key to navigate instead of their mouse
  • DON'T FIX IT WITH TABINDEX, do it right the first time!

Tab Order

DISCLAIMER: If you have to play with tab order, you may want to rethink your structure.

  • The tabindex property customizes the tab and navigation order when using the tab key on the keyboard
  • You can order elements using tabindex=1, tabindex=2, tabindex=3, and so on (But this is really bad! Don't do it!!)
  • A tabindex of 0 adds an element such as a <div> or <span> into the default tab order and makes it focusable.
  • A tabindex of -1 allows focus to be set programmatically to elements (such as a modal), but takes the object out of the default tab order.

Don't forget your TH's!

Scope is your friend!

For EXTREME complex tables: you can use the headers property on a <TD> element, paired with the <TH> IDs of the corresponding headers. Scope works with rowspan & colspan, so this is rarely necessary.

Last Name Areskog 
							
<table>
	<tbody>
		<tr>
			<th scope="row">First Name</th>
			<td>Ali</td>
		</tr>
		<tr>
			<th scope="row">Last Name</th>
			<td>Areskog</td>
		</tr>
	</tbody>
</table>

Headers

Properly nest headers! H1, H2, H3, etc have meaning beyond a style. Headers should be nested to indicate a specific reading order. The H1 header is often reserved for the name of the site, but this is not necessary with HTML5 markup.

The hgroup tag takes the header element with the highest rank (h1,h2, etc) to use for the document outline, and associates lesser-headings without cluttering things.

Practical Applications

Images

<img src="catsObviously.jpg" alt="An Adorable Fuzzy Cat" />
  • Don't forget your Alt text, it helps users with visual impairments, people using an RSS feed, and if your images don't load for whatever reason your description will.
  • Don't use images for functional elements such as Buttons. If you really really have to-- include a proper label and/or role.
  • Don't sweat the small stuff, decorative images and/or images with repetitive alt text should have a blank alt tag.

Hiding things

display: none; the element is not visible, is not rendered, and will not be seen or read by screenreaders

visibility: hidden; the element is not visible, it IS rendered, and will not be seen or read by screenreaders

aria-hidden HTML property hides from screenreaders only, visible and rendered for sighted users

Why is hiding things such an issue

Unfortunately, in many cases there are visual cues that indicate to sighted users what the hidden content or labels are that don't not get translated in a useful manner for those who are blind or visually impaired.

Some Examples: Standard dropdown navigation, tabs, accordions, complex inputs (addresses, phone numbers)

What can I do? I gotta hide it!

Hide these elements using a screenreader-friendly CSS class. Do not use display: none; or visibility: hidden;

The following example is taken from bootstrap, but there are tons of effective methods. HTML5 boilerplate's "visuallyhidden" method is also a great one. And it's creator explains it well.


.reader-only {
    position: absolute;
    width: 1px;
    height: 1px;
    padding: 0;
    margin: -1px;
    overflow: hidden;
    clip: rect(0,0,0,0);
    border: 0;
}

Forms & Labels: The Basics

All inputs, selects, and textareas should have a corresponding label. And the label's for property corresponds to the ID. * The "for" property is technically not necessary if the input is wrapped inside the label. It is considered implicit. *


							


	
	


<div>
	<label for="trust"><input type="checkbox" name="argueLong" id="trust" />You said you would do it yesterday, and I trusted you</label>
	<label for="listen"><input type="checkbox" name="argueLong" id="listen" />You never listen</label>
</div>

Labels, Round 2

So...you made an element without a visible label, or the label is on an unexpected area of the page.

WAI-ARIA to the rescue

If you really insist...

Use aria-label or aria-labelledby

This WILL NOT help your sighted users, but it will help those using screenreaders. I encourage a more universal approach!

aria-label is a label stand-in, and literally labels. When a screenreader reaches an aria-label it should alert your user

aria-labelledby points to the ID of other element(s) to label your form item
